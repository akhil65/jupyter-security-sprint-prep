🔐 Jupyter Security Sprint — Architecture

Two tools built for the Jupyter security ecosystem · March 2026

What's Being Built

RUNTIME

jupyter_sec_firewall

A Jupyter Server extension that sits between the browser and the Python kernel. Every time you run a cell, it intercepts the code before the kernel sees it, runs an AST security check, and either allows or blocks execution — showing an error in the notebook if blocked.

Runtime Protection WebSocket Intercept AST Analysis
CI / SPRINT TOOL

appsec_sprint_evaluator

A CLI tool that orchestrates a full security evaluation pipeline — running SAST (bandit/semgrep), SCA (pip-audit), secret detection, IaC scanning, DAST probing, and AI-assisted triage — then consolidates everything into a unified dashboard and optionally opens draft GitHub PRs with suggested fixes. Secrets, IaC, and AI-SPM stages are stubs; SAST and SCA are fully integrated.

What Each Stage Catches

SAST — Insecure Code Patterns
shell=True, XSS in templates, SQL injection strings, hardcoded creds in code
SCA — Vulnerable Dependencies
CVEs in pip packages, outdated tornado/jinja2/requests versions
Secrets — Leaked Credentials
API keys, tokens, passwords committed to git history or in source
IaC — Infrastructure Misconfig
Public S3 buckets, Dockerfiles running as root, open security groups
AI-SPM — Notebook Data Leaks
PII in cell outputs, model weights exposed, prompt injection in notebooks
DAST — Live Server Probe
Unauthenticated /api/kernels access, missing security headers (CSP, X-Content-Type-Options, X-Frame-Options)

Top Findings by Priority

HIGH · BANDIT B701 + SEMGREP
Jinja2 autoescape=False (XSS root cause)
serverapp.py:383 · app.py:3210, 3218
Jinja2 Environment created without autoescape=True. All templates inherit this — any user-controlled variable in HTML context is an XSS vector.
HIGH · BANDIT B602
subprocess(shell=True) in proxy.py
jupyterhub/proxy.py:757
Runtime code (not a test or build script) calls Popen with shell=True. If any part of the command is user-influenced, this is command injection.
HIGH · SEMGREP
Template vars in <script> tags (admin.html)
share/jupyterhub/templates/admin.html:5,6,8
Variables injected directly into JavaScript blocks using | safe. If any variable is user-controlled, this is stored XSS in the admin panel.
MEDIUM · SEMGREP
| safe filter on login.html (2 locations)
jupyter_server/templates/login.html:18,88
Autoescaping explicitly disabled on the login page. Requires careful audit of what data flows into these template variables.
MEDIUM · BANDIT B608
String-formatted SQL query
services/sessions/sessionmanager.py:411
SQL query constructed via string formatting. If any parameter is user-controlled without sanitization, SQL injection is possible.
MEDIUM · SEMGREP
Credential path logged
jupyter_server/auth/__main__.py:41
"password stored in config dir: %s" — password value written to logger output. Could expose credentials in log files.
MEDIUM · SEMGREP
Nginx $http_host in proxy docs
docs/howto/configuration/config-proxy.md
Documentation nginx examples use $http_host (attacker-controlled Host header). Operators copying these examples are vulnerable to Host header injection.
LOW · SEMGREP
Dockerfile runs as root
examples/service-fastapi/Dockerfile:13
No USER directive in example Dockerfile. Container runs as root. Example code gets copied into production deployments.

Recommended CI Strategy

Every PR
bandit skip B101
Fast, offline, catches new bad patterns as they land. Add .bandit config to skip assert_used.
On Dep Changes
pip-audit
Triggered when requirements.txt or pyproject.toml changes. Low noise, high value CVE data.
Weekly / Security PRs
semgrep
Full OWASP Top 10 sweep. Slower but thorough. SARIF output feeds GitHub Code Scanning.
